## Deviations & Open Questions

### 1. LOC.writer overrun (DEVIATED — material)
`src/sdlc/journal/writer.py` is **266 raw / 223 non-blank lines**, exceeding the spec's hard cap of "≤ 200 LOC" (Task 2 final bullet, spec L94). The spec also prescribes a specific remediation: "If it overruns, factor `_canonicalize_entry` and `_read_highest_seq` into `journal/_canonical.py` and `journal/_seq.py`" (spec L94). Neither file was created. The implementation's modularity is otherwise good (one helper per protocol step), but the spec invariant was missed. The Completion Notes (story L439) report quality gates green but never mention the LOC count. Recommendation: either split `_canonicalize_entry` + `_read_highest_seq` into sibling modules or get explicit waiver from PM.

### 2. AC2 property test structure: 4 properties bundled into 2 `@given` functions (PARTIAL)
Spec L46-52 enumerates four distinct properties:
- Property 1: file grows-only + bytes immutable (spec L46-49)
- Property 2: no-mutation API (spec L113 — module-level structural assertion, not `@given`)
- Property 3: iter_after correctness (spec L114, "for `k = N // 2`")
- Property 4: seq regression rejected (spec L115, "in a separate `@given`-decorated function")

The implementation has **only 2 `@given` functions**: `test_file_grows_only_and_line_bytes_immutable` (covers Properties 1 + 3 inline at lines 992-1001) and `test_seq_regression_rejected_and_file_unchanged` (Property 4). Property 2 is module-level structural and is correctly so. The bundling of P1+P3 into one function means each runs 1000 examples sharing fixtures rather than 4000 distinct hypothesis trials as the commit message claims ("4 hypothesis properties × 1000 examples"). Functional coverage is achieved; structural fidelity to spec L52 ("a separate `@given`-decorated function" applies only to P4, but P3 plausibly also intended its own) is partial. Open question: did the implementer intend P1+P3 bundle, or is this an inadvertent compression?

### 3. Property test seq strategy uses dict→model_validate, not direct st.builds (deviation from spec L110-111, but functionally equivalent)
Spec L110 prescribes `journal_entry_strategy = st.builds(JournalEntry, ...)`. Implementation uses `st.fixed_dictionaries({...})` then constructs entries via `JournalEntry.model_validate({**entry_dict, "monotonic_seq": seq})` (test_journal_append_only.py:902-942). Same final entries, different strategy plumbing. No correctness impact; flagging only because spec L110 is explicit about `st.builds`.

### 4. Property test does not enforce dict-payload string isprintable filter on values (minor)
Spec L110 specifies `payload=st.dictionaries(st.text(...).filter(str.isprintable), st.text(min_size=0, max_size=20), ...)` — the value strategy has no `isprintable` filter in the implementation (line 913: `st.text(min_size=0, max_size=20)`). Spec also did not require it on values, so this is consistent with spec L110 verbatim. Not a deviation.

### 5. No reader test for `iter_entries_works_on_windows` skip-on-Windows guard removed (spec L186)
Spec L186 says reader Windows test runs cross-platform via `pytest.mark.skipif(sys.platform == "win32", ...) REMOVED`. The implementation's reader test file has NO `pytestmark` skip (test_journal_reader.py:1341 just `pytestmark = pytest.mark.unit`), so it does run cross-platform. MET, but flagging that the spec phrasing is ambiguous about whether a `test_iter_entries_works_on_windows` named test should explicitly exist — it does (line 1451-1459).

### 6. Open question: bare-noqa interaction with valid violation line (spec L71)
The linter at check_no_journal_mutation.py:309-332 has two paths: `_filter_violations_by_noqa` (which re-emits "requires a reason" if violation lineno has bare-noqa) AND `_find_bare_noqa` (which scans all lines for bare-noqa not already in violation set). On a line that has BOTH a violation AND a bare-noqa, `_filter_violations_by_noqa` emits ONE entry; `_find_bare_noqa` skips it (existing_linenos contains it). Result: single "requires a reason" emitted. This appears correct and is tested at `test_filter_violations_bare_noqa_flagged`. Not a deviation.
