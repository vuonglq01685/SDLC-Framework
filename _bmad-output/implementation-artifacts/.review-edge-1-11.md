## BLOCKERS

### B1. `_read_highest_seq` ignores prefix that follows a torn final line — silent seq corruption window

- **Location:** `src/sdlc/journal/writer.py:_read_highest_seq` (lines ~571-601)
- **Trigger / scenario:** Process A (or a previous run, before the lock existed in this version) crashes mid-`os.write` having flushed e.g. `{"monotonic_seq": 99, "ts":"2026-...` without trailing `\n`. Process B acquires flock, calls `_read_highest_seq`. The reader uses Python's text-mode `for line in fh`, which yields the partial line as the *last* line; `JournalEntry.model_validate_json` raises `ValueError`; the function logs a warning and **continues iterating without bumping `highest`**. Because the reader treats malformed-as-skip and only tracks `highest = max(...)`, the partial entry's seq=99 is invisible. Now B writes its own seq=99 (which it picked from elsewhere as "next") on top, and the journal contains *one valid line at seq 50* + *garbage prefix* + *one valid line at seq 99*. Reader's monotonicity guard (`reader.py:466-467`) does NOT catch this because the bad line is skipped and seq order between the two surviving valid entries is still strictly increasing.
- **Observed vs expected:** Observed: validation passes, journal silently contains a torn-write artifact between two "valid" entries; downstream `iter_entries` returns 50, 99 with no error. Expected: torn final line should either (a) be detected and rejected, (b) cause writer to refuse to append until repair, or (c) be truncated atomically before append. ADR text claims O_APPEND atomicity prevents this, but `_read_highest_seq` opens with text mode `"r"` separately and does not validate "every line parsed cleanly."
- **Guard sketch:** `if any(skipped) and journal_path.stat().st_size > last_good_offset: raise JournalError(step="torn_journal_repair_required", ...)` — refuse to append on top of a torn tail; require explicit recovery.

### B2. `_read_highest_seq` reads an unbuffered file *outside* the kernel write barrier — stale read on relaxed-cache filesystems

- **Location:** `src/sdlc/journal/writer.py:_read_highest_seq` (line 582) opens with text-mode `"r"` while the *same process* will then `os.open(...O_APPEND)`. On filesystems with split read/write caching (some NFSv3, fuse passthroughs, certain virtio-9p mounts seen in CI containers), the text-mode handle's buffered read can see a stale view of a flushed-by-another-process append.
- **Trigger / scenario:** Two processes hold `journal.log.lock`. P1 finishes step-by-step append to seq=10 and releases flock. P2 acquires flock, calls `_read_highest_seq`. Filesystem returns cached metadata showing seq=9 as highest (P1's write fsynced data but P2's *open-for-read* hits a stat cache from before the write was visible). P2 writes seq=10; `O_APPEND` puts it at EOF after P1's seq=10. Two seq=10 entries on disk; reader's invariant guard (`<=`) catches it but it is a corruption that must not happen and the property tests do not exercise multi-process file-cache visibility (they run inside one process via `append_sync` only).
- **Observed vs expected:** Observed: silent duplicate seq committed; only caught at read-time. Expected: writer should `O_DIRECTORY|O_RDONLY` open the parent and fsync it, OR re-stat after acquiring flock to invalidate cache, OR open the journal with `os.O_RDONLY` directly and read via `os.read` (consistent with the `os.write` path). state/atomic.py performs parent-dir fsync precisely for this reason; journal/writer.py omits it (see comment lines 676-678) but the comment only argues correctness for *first-ever* O_CREAT — it does not address the read-side cache visibility problem.
- **Guard sketch:** `os.fsync(os.open(parent, os.O_RDONLY)) before _read_highest_seq` OR perform the seq scan via raw `os.open + os.read` so it shares cache coherence semantics with the upcoming `os.write`.

### B3. `_find_seek_then_write_violations` produces false negatives across nested function bodies — defeats AC3

- **Location:** `scripts/check_no_journal_mutation.py:217-279` and `_Visitor.visit_FunctionDef` (line 292).
- **Trigger / scenario:** The visitor explicitly calls `_find_seek_then_write_violations(node)` and then `self.generic_visit(node)`. `_collect_calls` walks the entire subtree (including nested defs). When `generic_visit` later descends into a nested `FunctionDef`, **the same seek/write pair is detected again** — duplicate report. Worse: a `seek` at the OUTER scope on `f` followed by a `write` on `f` *inside* an inner closure is reported (technically OK), but a `seek` *inside* an inner closure followed by a `write` at the OUTER scope is also reported (false positive). Conversely, when the outer body uses `f.seek(0)` *and then* defines an inner function whose first call is `g.write(...)` on a *different* receiver, the outer-scope `f.seek` will linger in `seen_seeks` and pair with any later `f.write` regardless of *control flow* (e.g., inside an unreachable `if False:` branch, in an exception handler that requires `f` to be re-bound, or after `f = open_a_different_file(...)` shadowing).
- **Observed vs expected:** Observed: writes that legitimately re-bind `f` to a fresh handle after a seek-only debug snippet flag false; nested defs double-flag. Expected: detector must scope per `FunctionDef` only (do not descend into nested defs), and must not pair `seek` with `write` across an intervening assignment to the same name.
- **Guard sketch:** In `_collect_calls`, skip nested `FunctionDef` / `AsyncFunctionDef` / `Lambda` subtrees; reset `seen_seeks` on `ast.Assign` to the same target name; in `_Visitor.visit_FunctionDef` do NOT call `self.generic_visit(node)` for the seek/write check (call only for the open/replace check via a separate visitor pass).

### B4. Linter ignores `seek+write` and `lseek+write` at module top level

- **Location:** `scripts/check_no_journal_mutation.py:282-300` (`_Visitor`).
- **Trigger / scenario:** `_find_seek_then_write_violations` is invoked **only** from `visit_FunctionDef` / `visit_AsyncFunctionDef`. A malicious or careless module-level script:
  ```python
  f = open("path/to/journal.log", "rb+")
  f.seek(0)
  f.write(b"overwrite")
  ```
  The `open(..., "rb+")` IS caught by `_check_open_call` (good), but if the file path is computed (e.g., `f = open(JOURNAL_PATH_VAR, mode_var)` where `mode_var` is dynamic), `_check_open_call` returns None (mode is not a Constant). The seek+write at module scope is then **invisible** because the visitor never analyzes module-level calls for this anti-pattern.
- **Observed vs expected:** Observed: module-level `f.seek(...) ; f.write(...)` is unflagged. Expected: AC3 says journal mutation outside writer.py must fail pre-commit regardless of scope.
- **Guard sketch:** Add `_find_seek_then_write_violations` over `tree.body` directly (treat the module like a function body) inside `_scan_file`.

### B5. `_NOQA_PATTERN` reason regex matches noqa text on unrelated lines, suppressing real violations

- **Location:** `scripts/check_no_journal_mutation.py:117` regex `r"#\s*noqa:\s*journal-mutation(?:\s*(?:—|--)\s*(.{10,}))?"` and `_filter_violations_by_noqa`.
- **Trigger / scenario:** A long string literal or docstring earlier in the file contains the substring `# noqa: journal-mutation -- documentation`. `_NOQA_PATTERN.search(line)` matches anywhere in the line, including inside a `"""..."""` docstring or a `r"# noqa: journal-mutation -- ..."` test fixture string. If a real violation occurs on the same logical line as a docstring continuation, the noqa silently silences it.
  More concretely: `"""Banned: open('journal.log', 'w')  # noqa: journal-mutation -- example"""` on a single line — `_check_open_call` doesn't fire (it's a string literal), but the same docstring discussing seek+write semantics on the same line as a real `f.seek(0)` would silently suppress a real seek+write detection 2 lines later if that real violation happens to be on the line containing the docstring closing quotes.
- **Observed vs expected:** Observed: noqa is line-scoped textually only, with no AST-anchored verification that the noqa is on the actual statement line. Expected: noqa should match only when the comment is on the same source line as the offending Call's `lineno`, AND the linter should ignore noqa-pattern matches inside string constants.
- **Guard sketch:** Strip strings/docstrings before searching; or precompute a set of `lineno` belonging to AST string literals and exclude them.

### B6. Linter fails to detect any of: `os.open(journal_path, O_WRONLY|O_TRUNC)`, `truncate()`, `ftruncate()`, `os.unlink(journal_path)`, hard-link / symlink replacement

- **Location:** `scripts/check_no_journal_mutation.py:_Visitor` — only checks `open` / `Path.write_text|write_bytes` / `os.replace|rename` / `seek+write` / `lseek+write`.
- **Trigger / scenario:** Outside `writer.py`, code can:
  - `os.open(journal_path, os.O_WRONLY | os.O_TRUNC)` — bypasses every check (not builtin `open`, not `os.replace`); then `os.write(fd, b"")`.
  - `Path(journal_path).unlink(); Path(journal_path).touch()` — wipes journal.
  - `pathlib.Path(journal_path).open("w")` — Path's `.open(mode="w")` method. `_check_open_call` only matches `Name id="open"` or `Attribute attr="open"`. `Path.open` IS Attribute attr="open", so it matches. BUT the first arg in `path_obj.open("w")` is the *mode* — `node.args[0]` is the literal string `"w"`, not a journal path. `_check_open_call` checks `node.args[0]` for journal-path expression and `node.args[1]` for mode — both shifted by one. Detector exits via `len(node.args) < 2`. Bypass succeeds. (The fixture file claims to test "Path.write_text" but never tests `journal_path.open("w")` style.)
  - `os.truncate(journal_path, 0)` and `f.truncate()` after a seek — entirely unhandled.
  - `os.link(other, journal_path)` / `os.symlink(other, journal_path)` — replaces journal with a link.
- **Observed vs expected:** Observed: AC3 ("journal mutation outside writer.py must fail pre-commit") is incomplete; multiple practical bypass paths exist. Expected: at minimum cover `os.open` with write flags, `Path(...).open` method form, `truncate`, `unlink`, `link`, `symlink` against journal-path expressions.
- **Guard sketch:** Add `_check_os_open_call`, `_check_path_open_call` (recognize `<expr>.open(<mode>)` where `<expr>` matches `_is_journal_path_expr`), `_check_truncate_call`, `_check_unlink_call`, `_check_link_call`.

### B7. Property test `test_file_grows_only_and_line_bytes_immutable` re-canonicalizes from in-memory entries — does NOT actually verify on-disk byte immutability

- **Location:** `tests/property/test_journal_append_only.py:952-1001`.
- **Trigger / scenario:** The "byte immutable" assertion compares `_canonicalize_entry(entries[j])` (recomputed each iteration) against `(raw_lines[j] + "\n").encode("utf-8")`. Both are derived from the SAME in-memory `entries[j]` and the SAME deterministic canonicalizer. If `_canonicalize_entry` is non-deterministic (e.g., a future change introducing a timestamp, hash salt, or relying on dict-ordering on a non-CPython interpreter), the test would still pass because both sides re-derive from `entries[j]`. The "immutable on disk" property requires capturing the bytes at write time `i` and re-checking them at write time `i+k` — which the test does NOT do. It only checks "current line bytes match what canonicalize_entry returns now."
- **Observed vs expected:** Observed: property gives a false sense of "line bytes immutable across appends" — actually only proves "current re-canonicalization equals current file content." A bug that *rewrites* line j when appending j+1 (e.g., if writer accidentally uses O_TRUNC) would pass IF the rewrite happened to produce the same canonical bytes (likely with this canonicalizer). It would only catch a rewrite that produced *different* bytes.
- **Guard sketch:** Snapshot `bytes_at_step_i = journal_path.read_bytes()` after each append, then on every subsequent step `assert journal_path.read_bytes().startswith(bytes_at_step_i)`. This is the actual append-only invariant.

### B8. `append_sync` event-loop guard is bypassable from a thread inside an asyncio program

- **Location:** `src/sdlc/journal/writer.py:742-754`.
- **Trigger / scenario:** `asyncio.get_running_loop()` is per-thread. If production code calls `append_sync` from a thread spawned via `loop.run_in_executor` or `asyncio.to_thread`, no loop is running in that thread → guard passes silently. The function's docstring claims "do NOT call from production code paths" and the guard is described as a "footgun guard mirroring state.atomic.write_state_atomic_sync" — but the guard does not actually protect against the dominant misuse pattern (calling sync API from inside an async program via a worker thread). This same gap exists in `state/atomic.py`, so it is consistent drift but still a real edge case for this story.
- **Observed vs expected:** Observed: a sync append called from an executor blocks the executor thread for fsync + flock — expected behavior, but the explicit guard advertises stronger protection than it provides. If `append_sync` is later "enforced for tests only" the assumption breaks.
- **Guard sketch:** Document precisely OR check `threading.main_thread() is threading.current_thread()` AND `asyncio.get_event_loop_policy().get_event_loop()` is closed.

### B9. `iter_entries` `OSError` handler swallows `OSError` from inside the generator, but the `with` block's `__exit__` may itself raise — handler does not cover that

- **Location:** `src/sdlc/journal/reader.py:450-486`.
- **Trigger / scenario:** Generator is partially consumed; consumer breaks out (e.g., `for e in iter_entries(p): break`). At GC time the generator's `with journal_path.open(...) as fh` calls `fh.__exit__(None, None, None)`. If close raises `OSError` (rare but possible: `EIO` on flush, NFS stale handle), the error escapes uncaught from the `__exit__` because the surrounding `try/except OSError` block has already exited normally on the prior yield. The generator raises raw `OSError` instead of the `JournalError` wrapper documented in the module's contract.
- **Observed vs expected:** Observed: callers expecting `JournalError` for all I/O failures see a bare `OSError`. Expected: any I/O failure during read should be wrapped consistently.
- **Guard sketch:** Wrap the `with` block + iteration inside a `try/except OSError` at the outer scope (the current `except OSError` only sees errors raised between `enter` and the first `yield`'s resume; close-on-GC errors slip past).

### B10. Writer omits parent-directory fsync after first-ever `O_CREAT` — first journal entry is durably-written to inode but the inode's directory entry is NOT durably linked

- **Location:** `src/sdlc/journal/writer.py:670-679` (comment "no parent-dir fsync").
- **Trigger / scenario:** The journal file does not exist. `_open_journal_for_append` does `O_CREAT | O_WRONLY | O_APPEND`, write succeeds, fsync(fd) succeeds, close succeeds. ADR-014 / the comment claims "only the first-ever O_CREAT creates a new directory entry (v1 accepted gap)." A power loss between `os.fsync(fd)` and an implicit later parent-dir fsync (which never happens) leaves a journal whose data blocks are durable but whose directory entry may not be — i.e., the journal file *vanishes* entirely on reboot, even though `append_sync` returned success. `state/atomic.py` *does* fsync the parent dir for exactly this reason (line 119-134). For a *journal* whose entire purpose is durability, accepting this gap contradicts NFR-REL-2.
- **Observed vs expected:** Observed: first-ever entry is at risk of vanishing entirely after kernel-panic / power loss between writer return and reboot. Expected: parent-dir fsync at least on the create path. The diff explicitly chooses not to do it.
- **Guard sketch:** `if not journal_path.existed_before_open: _fsync_parent_dir(...)` — same pattern as state/atomic.

## HIGH

### H1. `_canonicalize_entry` uses `ensure_ascii=False` — non-NFC-normalized JSON keys silently round-tripped if `_normalize_strings` misses a key

- **Location:** `src/sdlc/journal/writer.py:557-568`. `_normalize_strings` normalizes string *values* and dict *keys* (it does iterate `k` via `{k: ...}`). However Pydantic's `model_dump(mode="json")` may emit string keys that already came from `payload` dict (user-supplied). The recursion does NFC-normalize them, but on Python ≥3.7 dict ordering is insertion order — `sort_keys=True` saves us. Issue: if a user passes a `payload` whose key contains code points that NFC-decompose differently from NFD-decomposed input bytes that happened to be on disk previously, two distinct disk byte sequences can produce the same `entry` after NFC normalization. Coupled with the `_read_highest_seq` "highest only" check, a maliciously crafted prior entry whose monotonic_seq is, say, 100, and whose JSON parsed identically to a new entry the writer is about to emit, can produce two different on-disk byte sequences for what should be the same entry. The property test does NOT cover Unicode-equivalence at the byte level.
- **Trigger / scenario:** Hypothesis strategy filters via `str.isprintable` but does not filter against combining characters, surrogate halves, or BiDi marks. Some `isprintable() == True` strings include U+0301 combining accent. NFC-normalized output bytes will differ from raw bytes the user intended to provide, but that is by design — the actual edge case is a hand-crafted journal file where line N has NFD form and the writer's internal seq scan reads it, NFC-normalizes via `JournalEntry.model_validate_json`, producing seq metadata; but `entry.monotonic_seq` is purely numeric so this is benign. Real risk: `before_hash`/`after_hash` strings — if these are NFC-normalized, the hash chain integrity (Story 1.12) breaks because hashes are byte-sensitive but the writer just NFC'd a `sha256:xxx...` literal containing a stray combining mark snuck in via `actor` etc.
- **Observed vs expected:** Observed: `_normalize_strings` recurses into hash strings indiscriminately. Expected: hash fields and any opaque-bytes-in-string fields should be exempt from NFC normalization.
- **Guard sketch:** Pre-validate at JournalEntry contract level that `before_hash`/`after_hash` match `^sha256:[0-9a-f]{64}$`; OR add a sentinel set of field names to skip in `_normalize_strings` walking the dict.

### H2. `_JOURNAL_PATH_NAMES` regex misses common path expressions

- **Location:** `scripts/check_no_journal_mutation.py:120-123`.
- **Trigger / scenario:** Regex matches `journal.log`, `journal.jsonl`, `journal_path`, `JOURNAL_(PATH|FILE|LOG|JSONL)`. It does NOT match:
  - `cfg.journal` (attribute access naming convention)
  - `paths.journal_file_path` (matches due to `journal_path` containment? Yes, via substring — actually it does match because `journal_file_path` contains `journal_path`? No, `journal_file_path` does NOT contain `journal_path` as a substring. It contains `journal_file_path` only.)
  - `audit_log` / `audit.log` (alternate naming)
  - `state.journal_log` / `JournalPath` (PascalCase)
  - any variable named simply `j` or `log_file` after `j = config.journal_log`
- **Observed vs expected:** Observed: trivial rename of variable bypasses linter. Expected: stronger heuristic, e.g., dataflow from a known constant or detection of file extension `.jsonl` regardless of variable name. AC3 says outside writer.py must fail; today the heuristic is shallow.
- **Guard sketch:** Match any expression whose textual form contains `.jsonl` or `journal` (case-insensitive) anywhere; or scan strings for literal `*.journal.log` / `*.jsonl` filename patterns.

### H3. Property test does not cover `monotonic_seq` overflow / negative / zero edge cases

- **Location:** `tests/property/test_journal_append_only.py:_make_entry_sequence_strategy` (lines 920-941). Offsets ∈ [1, 10]; seq starts at 0 and grows. Final seq ≤ 50 × 10 = 500.
- **Missing scenarios:**
  - **seq = 0 immediately followed by seq = 0** (writer should reject — covered by `test_seq_regression_rejected_and_file_unchanged` but not in the multi-entry property).
  - **seq at `2**63 - 1`** (sys.maxsize): `_read_highest_seq` returns `int`; Python `int` is unbounded, so no overflow at runtime — BUT the JSON canonicalization writes the integer literal, and downstream consumers (Story 1.12 projection, possible JS clients) may overflow. The contract `Field(ge=0)` has no upper bound; appending seq = 2**100 is accepted today.
  - **seq = -1 sentinel collision:** `_read_highest_seq` returns `-1` for empty/missing. The writer compares `seq <= highest`. If a future contract change allows `seq = -1` (e.g., a "bootstrap" entry), the empty-file detection silently merges with a real entry.
  - **gap > 10:** strategy max `gap=10` does not exercise large jumps (e.g., seq 0 → seq 10**18) where JSON parsers may differ.
- **Observed vs expected:** Observed: property test has narrow range, missing arithmetic-boundary classes. Expected: include `st.integers(min_value=0, max_value=2**63-1)` cases, seq=0 at start, max-jump cases.
- **Guard sketch:** Extend offsets strategy to `st.integers(min_value=1, max_value=2**62)` (occasional) plus an explicit `@example(...)` for `[0, 1, 2**63-1]`.

### H4. `iter_entries` skips blank lines silently, masking torn-write artifacts

- **Location:** `src/sdlc/journal/reader.py:454` (`if not stripped: continue`).
- **Trigger / scenario:** A torn write that produced only `\n` (kernel buffered the trailing newline before crash) yields a blank line. Reader skips silently, no warning, no error. The reader's monotonicity guard does not fire because no entry was parsed. Downstream replays a journal that "looks correct" but is missing entries. The `_read_highest_seq` writer scan also skips blank lines, so a torn-write of *just the newline of a real entry* (data flushed but newline absent) is followed silently by the next append using a stale `highest`.
- **Observed vs expected:** Observed: blank line = silent skip. Expected: blank line in middle of journal should at least emit a warning; ideally the writer should detect "last byte is not `\n`" and refuse to append.
- **Guard sketch:** Before append, `os.fstat(fd).st_size` and check `pread(fd, 1, st_size-1) == b"\n"` if `st_size > 0`; refuse otherwise.

### H5. Reader does not validate that file ends with `\n` — final entry without newline silently dropped

- **Location:** `src/sdlc/journal/reader.py:451-479`.
- **Trigger / scenario:** A journal whose last entry is `{...}` without trailing `\n` (legitimate-but-incomplete write that crashed AFTER `os.write` returned the full payload's first chunk and BEFORE the `+ b"\n"` portion was issued — but the writer's `_canonicalize_entry` returns the bytes with `\n` already concatenated, so this requires an `os.write` short return that left exactly the trailing `\n` unwritten and then an exception was raised by the next `os.write` call which would have completed it). Reader's `for line in fh` returns the final line *without* trailing newline; `model_validate_json` succeeds (JSON parser doesn't care); entry is yielded as if valid. Next appender's `_read_highest_seq` does the same, reads the seq, and appends a new entry — producing `entry_N{...}entry_N+1{...}\n` glued together with no newline between them. Now reader sees a single line containing two JSON objects → `model_validate_json` fails → entire latter-half corruption is "skipped with warning."
- **Observed vs expected:** Observed: missing-trailing-newline corruption silently propagates and ultimately destroys the audit chain. Expected: writer must verify file ends with `\n` (or is empty) before appending; if not, raise `step="incomplete_terminator"`.
- **Guard sketch:** In `_append_protocol_body` after acquiring the lock and before `_open_journal_for_append`, `if journal_path.exists() and journal_path.stat().st_size > 0: pread last byte, assert == b"\n" else raise JournalError(step="terminator_missing")`.

### H6. Lock path collision / drift between `journal.log` and `journal.log.lock` — if user passes a path ending in `.lock`, the lock IS the journal

- **Location:** `src/sdlc/journal/writer.py:730, 766`. `lock_path = journal_path.with_suffix(journal_path.suffix + JOURNAL_LOCK_SUFFIX)`.
- **Trigger / scenario:** `journal_path = Path("/var/log/audit.lock")` (legitimate name choice or user typo). `lock_path = Path("/var/log/audit.lock.lock")`. Inverse case: `journal_path = Path("/var/log/journal")` (no suffix). `with_suffix(".lock")` on a path with empty suffix → `Path("/var/log/journal.lock")` — lock is sibling, fine. But `journal_path = Path("/var/log/")` (trailing slash → no name) raises `ValueError` from `with_suffix`. The async/sync wrappers do not catch this; the user gets a raw `ValueError`, not a `JournalError(step="validate_path")`.
- **Observed vs expected:** Observed: malformed paths raise `ValueError` instead of `JournalError`. Expected: validate `journal_path.name != ""` before `with_suffix`.
- **Guard sketch:** `if not journal_path.name or journal_path.suffix == JOURNAL_LOCK_SUFFIX: raise JournalError(step="validate_path", ...)`.

### H7. `_filter_violations_by_noqa` mishandles violation lineno > len(lines)

- **Location:** `scripts/check_no_journal_mutation.py:312-322`. Defensive `line = lines[lineno - 1] if lineno <= len(lines) else ""` is correct, BUT `splitlines()` strips the trailing newline. If the source ends without a final newline, `len(lines)` may be off-by-one relative to AST `lineno` for the *last* statement. Empty file edge case: `lines = []`, AST tree is empty → no violations → no issue. Edge case: a single-line file with `open("journal.log","w")  # noqa: journal-mutation -- reason` and NO trailing newline: `splitlines()` returns `[<line>]`, `lineno = 1`, OK.
- **Trigger / scenario:** A file ending with a `\` line-continuation joining two physical lines into one logical AST line — `lineno` points to the second physical line, but `splitlines()` returns both as separate strings; the noqa on the *first* physical line would not be seen by this lookup.
- **Observed vs expected:** Observed: noqa on continuation-line code is missed. Expected: also check `lines[lineno - 2]` for continuation cases, or use `tokenize` instead of `splitlines`.
- **Guard sketch:** Look back across `\`-continued lines; or canonicalize via `ast.parse`'s `end_lineno`.

### H8. `_find_lseek_then_write_violations` confuses `os.lseek(fd, 0, 0)` with another module's `os.lseek` after `import os as _os`

- **Location:** `scripts/check_no_journal_mutation.py:267`. Hard-coded `func.value.id == "os"` check.
- **Trigger / scenario:** `import os as _os` then `_os.lseek(fd, 0, 0); _os.write(fd, b"x")` — both checks see `func.value.id == "_os"`, so neither lseek nor write detection fires. Bypass complete.
- **Observed vs expected:** Observed: trivial rename of import alias bypasses the detector. Expected: track the alias of `os` per-file (via `ast.Import` aliases).
- **Guard sketch:** Pre-walk `tree.body` for `ast.Import` whose `name == "os"`, collect `aliases`, and check `func.value.id in os_aliases`.

### M1. `os.write` returning negative is theoretically impossible per CPython contract but defensive `if written == 0` does not catch negative

- **Location:** `src/sdlc/journal/writer.py:638`. `if written == 0:` — but a buggy mock or future stdlib change returning `-1` would loop with `offset += -1` forever or until OS error.

### M2. `_open_journal_for_append` perms mode `0o644` ignored when file already exists; but user's umask affects fresh creation
