## Story

As a **user enforcing phase-gate discipline in auto-mode**,
I want **the loop to halt when a phase signoff is required and not yet recorded (state machine = `awaiting-signoff` or `drafted-not-approved`), surfacing which phase needs signoff and the path to `SIGNOFF.md`**,
so that **auto-mode never silently advances past an unapproved phase** (PRD **FR21** trigger 2 [prd.md:764]; the loop's resume/perf contracts **NFR-REL-5** [prd.md:840] + **NFR-PERF-6** [prd.md:830] are inherited from 4.1 and must be preserved).

**AC1 — Positive trigger: an unapproved prior phase halts the loop (FR21 trigger 2).** *(epics.md:2080–2083)*
**Given** the auto-loop attempting to advance past a phase boundary,
**When** the next phase's first write is attempted and the prior phase's signoff state is `awaiting-signoff` OR `drafted-not-approved`,
**Then** the STOP-check detects it and halts the loop with `StopDecision(fired=True, trigger="signoff_required", target=<SIGNOFF.md path>, reason=<phase + state message>)` (C3 — phase packed into target/reason, not a StopDecision field),
**And** the journal records `kind=stop_triggered, trigger=signoff_required, target=<path>` via the **frozen** generic fired-branch (C2 — REUSE `stop_triggered`, no new kind),
**And** `state.json` reflects `auto_loop_status: halted, stop_reason: signoff_required` (via the **frozen** projection fold — zero projection edits).

**AC2 — Negative case: prior phase `approved` (valid hashes) → continue.** *(epics.md:2085–2088)*
**Given** the prior phase's signoff state is `approved` (canonical record present, `invalidated_at` null — `compute_state` returns `SignoffState.APPROVED`),
**When** the loop iterates,
**Then** STOP-check for trigger 2 returns `StopDecision(fired=False)`,
**And** the loop continues to the next phase / ready item (no `stop_triggered` entry, no halt). A **missing** `.claude/state/signoffs/` directory on a greenfield project is handled by `compute_state` (returns `AWAITING_SIGNOFF`, never an error) — see **D3** for why that must not spuriously halt before any boundary is crossed.

**AC3 — Resume: signed → loop continues (preserves NFR-REL-5).** *(epics.md:2090–2092)*
**Given** the loop halted on this trigger and the user signs the phase (Story 2A.12 — drafts `SIGNOFF.md` with `approved: true`, then `sdlc scan` writes the canonical record),
**When** I re-run `/sdlc-auto`,
**Then** the loop resumes; `compute_state(phase)` now returns `APPROVED`; STOP-check for trigger 2 returns `fired=False`,
**And** processing continues from the disk state at halt time (pure-function-of-disk — the resume reads the signed filesystem, no in-memory continuation).

### Signal source — what the trigger reads (and why it does NOT re-parse disk)

- **Read signoff state via `compute_state`, not raw YAML.** `engine → signoff` is an **allowed** dependency (`module_boundary_table.py:104` — `engine.depends_on` includes `signoff`), and `next_selector.py:13` already imports `compute_state` from `engine/`. So 4.3 reuses the canonical 4-state machine instead of pure-disk-parsing `.claude/state/signoffs/` — the prompt's "if engine→signoff disallowed, pure-disk read" branch does **not** apply (verified allowed). `compute_state` already handles the missing-directory greenfield case (returns `AWAITING_SIGNOFF`, never an error — `states.py:82–100`).
- **Phase-3 has no signoff** (`states.py:24,71–80`). The trigger evaluates phase-1 and phase-2 only. `compute_state(3, strict=False)` returns `AWAITING_SIGNOFF` + a once-per-process WARN — do not call it for phase 3.
- **Auto-loop STOP vs phase-gate HOOK — keep them distinct.** The phase-gate hook (`phase_gate.py`) blocks a *single write* to a phase-2/3 path via a `HookDecision.deny` at dispatch time. 4.3's STOP *halts the whole loop* and *surfaces to the user* via the `stop_triggered` journal entry + `halted` projection. Same signal (`compute_state != approved`), two different enforcement layers — the hook is per-write defense-in-depth; the STOP is the user-facing auto-mode halt FR21 demands. 4.3 touches neither the hook nor `next_selector`.

### The post-dispatch-only contract — read before implementing (C5)

`check_stop` runs only after a real dispatch (`auto_loop.py:286`), and `resolve_next_action` short-circuits to `run_command` (no dispatch) the moment phase 1 or 2 is unapproved (`next_selector.py:158–159,169–170`). So in the *literal* current loop, a project blocked at an unapproved phase 1 exits at `auto_loop.py:241` and never reaches 4.3's trigger. **This is by design and is NOT 4.3's to fix** — fixing it means editing the frozen 4.1 post-dispatch contract + the frozen 4.2 seam (out of C1 scope). 4.3 delivers a **correct pure-disk predicate** (`check()`), proven authoritatively by the unit test, and integration-tested through a project where dispatch *does* occur while a downstream phase boundary is unsigned (D3 — the recommended evaluation point lets a phase-3 task dispatch coexist with, e.g., an unsigned/invalidated phase-2 boundary; pick the D3 framing that makes the 4-cell matrix exercise `check_stop` realistically). If D3's realistic dispatch+unsigned coexistence proves impossible in the v1 loop without a no-work-path `check_stop` call, **escalate as a decision** rather than silently editing `auto_loop.py`.

## Decisions Needed

- **D1 — Encoding the phase + SIGNOFF.md path into the 4-field `StopDecision` (the epics names `phase=<N>`, but the field set is frozen).** `StopDecision` has only `{fired, trigger, target, reason}` (C3); the loop forwards only `trigger/target/reason` into the journal. The phase number + the path the user must edit have to live in those fields.
  - **(a) `target` = repo-relative `SIGNOFF.md` draft path (`PHASE_DIR_MAP[phase] + "/SIGNOFF.md"`, e.g. `01-Requirement/SIGNOFF.md`); `reason` = `"phase <N> signoff required (state=<awaiting-signoff|drafted-not-approved>); sign via /sdlc-signoff <N>"`.** `target` is exactly "the path to SIGNOFF.md" the AC demands and what 5.19's banner renders as the target id; `reason` carries the phase + the actionable next step. The phase number is recoverable from both. Reuse `signoff.PHASE_DIR_MAP` (no hardcoded dir names). Mirrors 4.2's `target = <file path>` convention. **(Recommended.)**
  - **(b) `target = f"phase-{N}"`; `reason` carries the SIGNOFF.md path.** Symmetric with the epics' `phase=<N>` phrasing, but loses the path as the primary `target` (5.19 renders `target` as the banner's target id, and the AC's headline ask is "the path to SIGNOFF.md") — demotes the more useful datum.
  - **(c) Pack both into `reason` as structured text; leave `target` = the path.** Same as (a) for `target`; differs only in `reason` verbosity. Fine, but keep `reason` human-readable, not a serialized blob (no JSON-in-a-string — that invites a downstream parser against an un-frozen shape).

- **D2 — Which signoff states halt (and the `invalidated-by-replan` boundary).** The AC names `awaiting-signoff` OR `drafted-not-approved` (epics.md:2081). The 4-state machine has a fourth state, `INVALIDATED_BY_REPLAN`.
  - **(a) Halt on exactly `{AWAITING_SIGNOFF, DRAFTED_NOT_APPROVED}`; treat `APPROVED` and `INVALIDATED_BY_REPLAN` as NOT-fired for trigger 2.** `INVALIDATED_BY_REPLAN` is the replan-dirty surface owned by **STOP trigger 4 (Story 4.5)** (epics.md:2124–2147 — "items marked dirty by `sdlc replan`"); folding it into trigger 2 would double-halt and blur the trigger taxonomy. Match the AC literally. **(Recommended.)**
  - **(b) Also halt on `INVALIDATED_BY_REPLAN`.** Over-broad — steps on 4.5's replan-dirty trigger and contradicts the AC's explicit two-state list. Rejected unless a panel reviewer wants trigger 2 to subsume invalidation (then coordinate with 4.5's owner).

- **D3 — Which phase boundary `check()` evaluates (and how it coexists with a real dispatch given the post-dispatch-only check, C5).** `State.phase` is always 1 (C8), so the phase must be derived from signoff records. The check must fire "when advancing PAST a boundary" while a dispatch actually occurred (so `check_stop` is reached, C5).
  - **(a) Evaluate the prior phase of the next-not-yet-approved phase: compute `compute_state(1)`; if not `APPROVED`, the boundary in question is phase 1 (fire on awaiting/drafted). Else compute `compute_state(2)`; if not `APPROVED`, the boundary is phase 2 (fire). Else not-fired (both approved → no signoff gate).** Mirrors `next_selector`'s ladder (`next_selector.py:150–171`) so the trigger's notion of "the relevant boundary" stays consistent with how the loop selects work. `target`/`reason` name that phase. **(Recommended.)** Integration framing: arrange a project where phase 1 is `approved` and phase 2 is `awaiting-signoff`/`drafted-not-approved` but a dispatch still occurs in the iteration (e.g. via the loop's dispatch path for an item that is ready), so `check_stop` is reached post-dispatch and fires on phase 2 — if the v1 loop cannot produce that coexistence (because `next_selector` short-circuits to `run_command` first, C5), drive the positive integration cell through `check_stop` directly (as `test_stop_clarification.py` proves the trigger via the loop only when a dispatch path exists) and rely on the unit test as the authoritative `check()` behavior surface. **Escalate** (do not edit `auto_loop.py`) if a realistic loop-level positive cell is unreachable.
  - **(b) Always evaluate phase 2 (the last signoff boundary before phase-3 implementation).** Simpler, but mis-reports when phase 1 is the unsigned boundary (reason would name the wrong phase) and diverges from `next_selector`'s ladder. Weaker.
  - **(c) Fire if ANY of phases {1,2} is in a halting state; `target`/`reason` name the lowest such phase.** Equivalent to (a) in the lowest-unsigned case; (a) is the precise framing. Acceptable if (a)'s "prior phase of next-not-approved" wording proves awkward to implement — they converge on the same lexically-lowest unsigned phase.
