## Story

As a **user wanting auto-mode to surface persistent failures rather than retry forever**,
I want **the loop to halt when an agent dispatch fails 3 times (1 attempt + 2 retries per Story 2A.3) on the same target**,
so that **I investigate root cause instead of burning retries** (PRD **FR21** trigger 5; the loop's resume/perf contracts **NFR-REL-5** + **NFR-PERF-6** are inherited from 4.1 and must be preserved).

>
> **(C1) SCOPE — 4.6 READS a pre-existing JOURNAL seam; it does NOT poll a `.claude/state/<thing>` file and does NOT re-journal the failure history.** Every other Layer-2 trigger (4.2 clarification, 4.7 high-risk-path, …) detects a *fresh disk artifact*. 4.6 is the odd one out: the signal it needs **already exists in the journal**. The 2A.3 dispatcher, on terminal dispatch failure (after `with_retries` exhausts `_max_attempts=3`), writes a `kind=stop_trigger_raised` placeholder entry via `dispatcher/_panel_helpers.py:_emit_stop_trigger` (called from `dispatcher/core.py:232/370/422/469`), and writes one `kind=dispatch_attempt` row per attempt (`outcome` ∈ `retry`,`retry`,`failed`). So **4.6's `AgentFailedTrigger.check()` reads recent journal entries** (via `iter_entries` — `engine` MAY import `journal`, boundary-verified `module_boundary_table.py:97–115`) to detect that an agent exhausted its retries, and returns `StopDecision(fired=True, trigger="agent_failed", target=<id>, reason=<encoded>)`. The loop's existing fired-branch then emits the `stop_triggered(trigger=agent_failed)` halt entry. **The "journal records the full failure history" AC clause is ALREADY satisfied by the dispatcher** (the `dispatch_attempt` rows + the `stop_trigger_raised` entry) — 4.6 **references** that history, it does **not** re-write it.
>
> **(C2) SHARED-FILE — 4.6 appends ONE line to `_ORDERED_TRIGGERS`; this is the merge point with siblings 4.3–4.8.** 4.2 froze the registry seam (`src/sdlc/engine/stop_registry.py`): `_ORDERED_TRIGGERS: tuple[StopTrigger, ...]` (`:13`) is a literal tuple, and the priority-ordering comment (`:11–12`) says *"Stories 4.3-4.9 append new triggers to this tuple after review."* 4.6 adds `AgentFailedTrigger()` to that tuple — **the only shared-file edit**. Because 4.3–4.8 each touch the same line in parallel worktrees, this is a guaranteed conflict point: **rebase before merge** (DAG §3 linear-merge discipline), and keep the trigger class itself in its own file (`stop_agent_failed.py`, zero body contention). Do **not** edit the frozen public surface of `stop_triggers.py` (the `StopDecision` field set `:16–23`, the `StopTrigger` Protocol `:26–32`, the `check_stop`/`register_stop_trigger` signatures `:38–45`) — they are frozen for Layer 2.
>
> **(C3) TWO DISTINCT KINDS — `stop_trigger_raised` (INPUT, read by 4.6) vs `stop_triggered` (OUTPUT, emitted by the loop). Do not conflate them.** This is the single most likely confusion in this story.
> - **`stop_trigger_raised`** = the **2A.3 INPUT signal 4.6 consumes**. Written by the *dispatcher* on retry-exhaustion. Its **real** payload (verified at `_panel_helpers.py:237–244`) is `{"trigger": "agent_failure_after_retries", "specialist": <name>, "step": <step_name>, "epic_4_placeholder": True}` plus optional `"last_error": str(exc)`; `target_id` is `f"{step_name}/{specialist_name}"`. **⚠ ADR-028 doc↔emitter mismatch:** the ADR-028 §3 taxonomy row for `stop_trigger_raised` (`docs/decisions/ADR-028-journal-kind-taxonomy.md:78`) documents the payload keys as `trigger_kind, reason, epic_4_placeholder` — but the emitter never writes `trigger_kind` or `reason`; it writes `trigger`, `specialist`, `step`, `epic_4_placeholder`, `last_error`. **4.6 must read the keys the emitter ACTUALLY writes (`specialist`, `step`, `last_error`, `trigger`), NOT the stale ADR keys.** (This is the same `trigger_kind` doc-drift 4.2's C3 warned about — `projection.py:76` only tolerates `trigger_kind` as a *fallback*.) Note that 4.6's `trigger_id` is **`agent_failed`** (per the AC), which differs from the placeholder's `trigger="agent_failure_after_retries"` string — the StopDecision the loop emits uses `agent_failed`.
> - **`stop_triggered`** = the **4.2 OUTPUT kind** the loop's fired-branch emits (`auto_loop.py:_make_stop_triggered_entry`, `:105–130`). 4.6 does **not** write this directly — `_finish_halted_on_stop_trigger` (`:153–178`) writes it generically from whatever `StopDecision` `check_stop` returns. **THE INTERACTION (real, surface as D2):** `projection.py:97–100` ALREADY folds **both** `stop_triggered` AND `stop_trigger_raised` → `halted` (and `_KNOWN_KINDS` `:40–51` lists both). So the dispatcher's pre-wired `stop_trigger_raised` placeholder, once on disk, *already* makes `project_from_journal` report `auto_loop_status="halted"` (reason = `agent_failure_after_retries`). 4.6's job is to make the **auto-loop** detect this and produce its own first-class `stop_triggered(trigger=agent_failed)` STOP via the registered-trigger path — so the loop halts *in-run* with the canonical `agent_failed` reason, rather than relying on a projection side-effect of a placeholder. 4.6 produces option **(a)** — detect retry-exhaustion from the journal, return a `StopDecision`, let the loop emit `stop_triggered(agent_failed)`. It does **NOT** reconcile/upgrade the placeholder's reason (option b) — that is out of scope and would couple the engine to dispatcher payload semantics.
>
> **(C4) ZERO new wire-format, ZERO new journal kind, ZERO projection edit — 4.6 is pure additive engine code.** This is the inverse of 4.2 (which had to add the `stop_triggered` kind + extend the projection fold). For 4.6 those are **already done**: `stop_triggered` exists (4.2), the projection fold maps it to `halted` (4.2), and the input kind `stop_trigger_raised` exists (2A.3). 4.6 introduces **no** new `JournalEntry.kind`, touches **no** `src/sdlc/contracts/`, regenerates **no** snapshot, and edits **no** `state/projection.py`. `freeze_wireformat_snapshots --check` stays **7/7**. The only new files are `engine/stop_agent_failed.py` + its two tests; the only modified file is `engine/stop_registry.py` (the one-line tuple append, C2). **Do NOT add an ADR-028 row** (no new kind). **Do NOT edit `auto_loop.py`** — its halt-finalizer is generic over `StopDecision` (verified `:153–178`); a vanilla trigger needs zero loop edits.
>
> **(C5) STOPDECISION HAS ONLY 4 FIELDS — pack `agent`/`attempts`/`last_error` into `reason` (surface as D1).** AC-1 wants the user to see `trigger=agent_failed, agent=<name>, target=<id>, attempts=3, last_error=<excerpt>`. But `StopDecision` is frozen at `{fired, trigger, target, reason}` (`stop_triggers.py:16–23`) and 4.2 froze it for Layer 2 (C2). **Do NOT add fields.** The mapping: `trigger="agent_failed"`; `target` = the failing target id (the `step`/`specialist` identifier from the placeholder's `target_id`, i.e. `f"{step}/{specialist}"`); the `agent` name + `attempts` count + `last_error` excerpt are **encoded into the free-string `reason`** (a human-readable one-liner, e.g. `agent=<specialist> attempts=3 last_error=<truncated excerpt>`). The **full** failure history (every attempt + the exact last_error) is **already in the journal** (C1) — `reason` is a *surfaced summary*, not the system of record. The exact `reason` encoding is **D1**.
>
> **(C6) NEGATIVE CELL — fail-then-succeed must NOT fire: a later success supersedes an earlier failure signal.** AC-2: an agent fails once then succeeds on retry → STOP false. In the journal this looks like `dispatch_attempt(retry) → dispatch_attempt(success)` and **no** `stop_trigger_raised` entry (the dispatcher only emits `stop_trigger_raised` after `with_retries` raises `DispatchError`, which only happens when the *last* attempt fails — `retry.py:55,84–93`, `core.py:231–238`). So the simplest, most robust detection is: **fire iff a `stop_trigger_raised` (terminal-failure) entry exists for the current dispatch that has not been superseded by a later success/resolution for the same target.** A fail-then-succeed run never writes `stop_trigger_raised`, so it never fires — clean by construction. **Reason carefully about scoping the journal read to the *current* dispatch** (use `correlation_id` and/or "most-recent entries since the last loop iteration") so a *stale* `stop_trigger_raised` from a prior, since-resolved dispatch on the same target does not re-fire (this is the AC-3 resume case — see C7/D2). Decide the supersession rule in **D2**.
>
> **(C7) RESUME (AC-3) — the dispatcher's retry counter resets on re-run; 4.6 must not re-fire on a stale signal.** When the user fixes the issue and re-runs `/sdlc-auto`, the loop re-dispatches; `with_retries` starts a fresh attempt sequence (the retry counter is per-dispatch, not persisted — `retry.py:73`). If the fix is real, the new dispatch writes `dispatch_attempt(success)` and **no** new `stop_trigger_raised`. 4.6's `check()` must therefore key off the **current** dispatch's outcome (latest entries / `correlation_id` of the iteration just dispatched), not merely "any `stop_trigger_raised` ever written" — otherwise a resolved failure would re-halt forever. The loop passes `check_stop` the **pre-dispatch** `state` snapshot (`auto_loop.py:286` is called *after* `dispatch_fn` at `:276`), and the journal is appended by the dispatch in between, so by the time `check()` runs the current dispatch's `dispatch_attempt`/`stop_trigger_raised` entries are already on disk — `check()` re-reads the journal directly (`_ = state`, mirroring 4.2's C7). Pin the supersession/scoping rule with a test (resume cell) and **D2**.
>
> **(C8) REAL-DISPATCH DEBT — test against the MOCK runtime with a STUB agent failing N times; do NOT wire real dispatch.** DAG §7 + ratified Decision **D3** (epic-4-dag §289–304): real `/sdlc-auto` dispatch is nominal-only / mock-only in places (`EPIC-3-DEBT-CHARACTERIZATION-REAL-DISPATCH` + `EPIC-2A-DEBT-TASK-REAL-TEST-EXECUTION` + `EPIC-4-DEBT-AUTO-REAL-DISPATCH`, all **registered in `debt-budget.yaml`** lines 126–139). The pre-emptive guard already ships: `cli/auto.py:53–61` raises `ERR_AUTO_LOOP_REAL_DISPATCH_DEFERRED` unless `SDLC_USE_MOCK_RUNTIME=1` (the Story 3.8 pattern) — **4.6 inherits this guard; it does NOT add a new one and does NOT register new debt** (already done). Tests run under `SDLC_USE_MOCK_RUNTIME=1` (autouse, `tests/conftest.py:39–48`). AC-4's "stub agent failing N times" = a dispatch stub that raises/returns failure N times so the dispatcher's `with_retries` exhausts and emits `stop_trigger_raised`, exercising 4.6's read path against journal entries the stub produced. Verify the stub-failure idiom against `tests/integration/test_dispatch_retry.py` (`runtime.dispatch.side_effect = DispatchError(...)` → 3 `dispatch_attempt` rows `[retry, retry, failed]`).
>
> **(C9) MODULE BOUNDARY + LOC + the dispatcher-internals boundary.** New code lives in `engine/` — `engine` MAY import `state`, `journal`, `ids`, `errors`, `config` (and `dispatcher`, but see the caveat) and MUST NOT import `cli`/`dashboard` (`module_boundary_table.py:97–115`; gate `scripts/check_module_boundaries.py`). **Boundary-aware constraint:** although `engine` *is technically allowed* to import `dispatcher`, 4.6 must read the **journal** (the durable seam), NOT reach into dispatcher in-memory state or call dispatcher functions — the contract is "engine reads the journal entries the dispatcher wrote," keeping the coupling at the audit-trail layer (this is what makes 4.6 testable with a stub and resilient to dispatcher refactors). **And** the surfaced decision carries `agent=<name>, target=<id>, attempts=3, last_error=<excerpt>` — encoded as `trigger="agent_failed"`, `target=<failing-target-id>`, and `reason=<one-line agent/attempts/last_error summary>` (C5 — no new `StopDecision` fields),
**And** the journal records the full failure history — **already** via the dispatcher's `dispatch_attempt` rows + the `stop_trigger_raised` entry (C1 — 4.6 references this, does not re-journal it) — plus the loop's `stop_triggered(trigger=agent_failed)` halt entry,
**And** `state.json` reflects `auto_loop_status: halted, stop_reason: agent_failed` (via the **existing** 4.2 projection fold — C4, no projection edit),
**And** the user is shown the path to the agent's last debug output (the agent-runs telemetry at `03-Implementation/agent_runs.jsonl` — `cli/auto.py:_RUNS_REL`; surface the path in `reason` or rely on the CLI/5.19 banner — see D1/D3).

**AC2 — Negative case: fail-then-succeed → continue (success cancels the retry chain).** *(epics.md:2163–2166)*
**Given** an agent that fails once then succeeds on retry,
**When** the dispatcher retries (writes `dispatch_attempt(retry)` then `dispatch_attempt(success)`, and **no** `stop_trigger_raised`),
**Then** STOP-check for trigger 5 returns `StopDecision(fired=False)` (C6 — the absence of a terminal `stop_trigger_raised` for the current dispatch makes non-firing automatic),
**And** the loop continues normally (no `stop_triggered` entry, no halt).

**AC3 — Resume: user fixes the issue, re-run → retry counter resets, loop continues (preserves NFR-REL-5).** *(epics.md:2168–2171)*
**Given** the loop halted on this trigger and the user fixes the underlying issue (updates a fixture, edits a prompt, restarts a flaky service),
**When** I re-run `/sdlc-auto`,
**Then** the dispatcher's retry counter resets for fresh attempts (per-dispatch, not persisted — C7),
**And** if the fix is real, the next dispatch writes `dispatch_attempt(success)` and STOP-check for trigger 5 now returns `fired=False` (the new success supersedes / is-scoped-past the stale failure — C7/D2),
**And** the loop continues from the disk state at re-run time (pure-function-of-disk — the resume reads the resolved journal, no in-memory continuation).

### The INPUT vs OUTPUT kinds — read this before implementing `check()`

- **`stop_trigger_raised` (INPUT, 2A.3, read by 4.6):** written by the dispatcher when `with_retries` exhausts. The placeholder carries `epic_4_placeholder=True` and a `TODO(epic-4)` marker (`_panel_helpers.py:235,241,249`; `core.py:7` module-docstring TODO; `deferred-work.md:51` `EPIC-4-STOP-TRIGGER-WIRE`). **This is the seam this story exists to consume.** Its `target_id` is `f"{step}/{specialist}"`, payload `trigger="agent_failure_after_retries"`, `specialist`, `step`, `last_error`. **The ADR-028 row (`:78`) lists stale keys `trigger_kind, reason` the emitter never writes — read the emitter's real keys (C3).**
- **`stop_triggered` (OUTPUT, 4.2, emitted by the loop):** the canonical halt kind. The loop's `_finish_halted_on_stop_trigger` writes it from 4.6's returned `StopDecision`; `trigger="agent_failed"`.
- **Why the trigger does its OWN detection (C3 interaction):** the projection *already* folds the dispatcher's `stop_trigger_raised` placeholder → `halted` (reason `agent_failure_after_retries`). But that is a passive projection side-effect with the wrong reason string and no in-run loop halt. 4.6 makes the **auto-loop** halt in-run with the canonical `agent_failed` reason via the registered-trigger path. 4.6 does **not** reconcile/upgrade the placeholder reason (C3 option b — out of scope).

### Detection + supersession (read before implementing D2)

The robust signal is **the presence of a terminal `stop_trigger_raised` for the current dispatch that has not been superseded by a later success/resolution on the same target.** Because the dispatcher writes `stop_trigger_raised` *only* on terminal failure (`retry.py:55` raises → `core.py:231` catches → `_emit_stop_trigger`), a fail-then-succeed run never produces it (C6) — so the negative cell is clean by construction. The subtle case is **AC-3 resume**: a *stale* `stop_trigger_raised` from a prior, since-resolved dispatch must not re-fire. Scope the read to the current dispatch — e.g. only consider `stop_trigger_raised` entries that are **more recent than the last `dispatch_attempt(success)` for the same `target_id`**, or that share the iteration's `correlation_id` / are the **most-recent** terminal-vs-success for that target. Prove both directions with the negative + resume cells. The exact rule is **D2**.

## Decisions Needed

- **D1 — `reason`-string encoding for `agent` / `attempts` / `last_error` + the debug-output path (C5).** `StopDecision` has only `{fired, trigger, target, reason}`; AC-1 wants `agent=<name>, attempts=3, last_error=<excerpt>` surfaced, and the user "shown the path to the agent's last debug output."
  - **(a) Pack a single human-readable one-liner into `reason`: `agent=<specialist> attempts=<N> last_error=<truncated excerpt> debug=03-Implementation/agent_runs.jsonl`.** `trigger="agent_failed"`, `target=<step/specialist>`. Truncate `last_error` to a fixed budget (e.g. 200 chars) to keep `reason` bounded. The **full** history stays in the journal (`dispatch_attempt` rows + `stop_trigger_raised.last_error`, C1); `reason` is a surfaced summary, not the system of record. The dashboard (5.19) and CLI can re-derive richer detail from the journal. Smallest surface, no new fields, satisfies every AC clause. **(Recommended.)**
  - **(b) Put only `agent=<name> attempts=<N>` in `reason`; rely on the CLI/5.19 banner to render `last_error` + debug path from the journal.** Cleaner `reason`, but AC-1's "last_error=<excerpt>" then has no representation in the StopDecision itself — pushes the obligation onto a not-yet-built consumer.
  - **(c) Encode a structured (e.g. `k=v;`-delimited or JSON) blob in `reason`.** Machine-parseable, but `reason` is a free human string elsewhere (4.2 used a bare trigger word), and a JSON-in-string invites brittle parsing; reject in favour of the human one-liner.

- **D2 — Supersession / scoping rule for the journal read (C6/C7 — makes the negative + resume cells pass).** `check()` must fire on a *current* terminal failure but NOT on a *stale, since-resolved* one.
  - **(a) Most-recent-wins per target: fire iff the latest terminal-relevant entry for the failing `target_id` is `stop_trigger_raised` (i.e. no later `dispatch_attempt(success)` for that target).** Scan `iter_entries` once, track per-`target_id` the last `dispatch_attempt(success)` seq vs the last `stop_trigger_raised` seq; fire on targets whose `stop_trigger_raised` is newer. Pure-fn-of-disk, deterministic (NFR-REL-5), handles resume (a real fix writes a newer success), and is robust to multiple targets. **(Recommended.)**
  - **(b) Correlation-id scoping: only consider entries sharing the current iteration's `correlation_id`.** Tightest scoping to the just-dispatched task, but the loop passes `check_stop` the pre-dispatch `state` (not the `correlation_id`), and `check()`'s Protocol doesn't receive it (frozen) — would require re-deriving the latest iteration's correlation_id from the journal anyway, so it collapses into (a) with extra coupling.
  - **(c) "Any `stop_trigger_raised` ever written fires."** Simplest, but **breaks AC-3 resume** (a resolved failure re-halts forever) and **breaks the multi-target case**. Rejected.

- **D3 — Priority placement of `AgentFailedTrigger` in `_ORDERED_TRIGGERS` (C2/C7 first-fired).** `check_stop` returns the first-fired decision; ordering only matters if two triggers fire in the same iteration.
  - **(a) Append after the clarification trigger (current tuple order: clarification, then agent_failed).** Clarification is the foundational STOP (4.2); agent-failure is a runtime/execution STOP. Co-firing is unlikely (a clarification halts before dispatch reaches retry-exhaustion), so order is largely cosmetic — but a stable, documented order is what the siblings inherit. Append-at-end matches the tuple-growth convention (`stop_registry.py:12`). **(Recommended.)**
  - **(b) Insert at a specific priority index** (e.g. before clarification). No evidence any STOP must out-prioritise another in v1; reordering the frozen tuple invites merge churn with 4.3–4.8. Defer any real priority contract to a dedicated story if a co-fire case ever surfaces. Reject pre-emptive reordering.
