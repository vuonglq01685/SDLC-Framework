## Story

As a **user wanting auto-mode to surface persistent failures rather than retry forever**,
I want **the loop to halt when an agent dispatch fails 3 times (1 attempt + 2 retries per Story 2A.3) on the same target**,
so that **I investigate root cause instead of burning retries** (PRD **FR21** trigger 5; the loop's resume/perf contracts **NFR-REL-5** + **NFR-PERF-6** are inherited from 4.1 and must be preserved).

### The INPUT vs OUTPUT kinds ‚Äî read this before implementing `check()`

- **`stop_trigger_raised` (INPUT, 2A.3, read by 4.6):** written by the dispatcher when `with_retries` exhausts. The placeholder carries `epic_4_placeholder=True` and a `TODO(epic-4)` marker (`_panel_helpers.py:235,241,249`; `core.py:7` module-docstring TODO; `deferred-work.md:51` `EPIC-4-STOP-TRIGGER-WIRE`). **This is the seam this story exists to consume.** Its `target_id` is `f"{step}/{specialist}"`, payload `trigger="agent_failure_after_retries"`, `specialist`, `step`, `last_error`. **The ADR-028 row (`:78`) lists stale keys `trigger_kind, reason` the emitter never writes ‚Äî read the emitter's real keys (C3).**
- **`stop_triggered` (OUTPUT, 4.2, emitted by the loop):** the canonical halt kind. The loop's `_finish_halted_on_stop_trigger` writes it from 4.6's returned `StopDecision`; `trigger="agent_failed"`.
- **Why the trigger does its OWN detection (C3 interaction):** the projection *already* folds the dispatcher's `stop_trigger_raised` placeholder ‚Üí `halted` (reason `agent_failure_after_retries`). But that is a passive projection side-effect with the wrong reason string and no in-run loop halt. 4.6 makes the **auto-loop** halt in-run with the canonical `agent_failed` reason via the registered-trigger path. 4.6 does **not** reconcile/upgrade the placeholder reason (C3 option b ‚Äî out of scope).

### Detection + supersession (read before implementing D2)

The robust signal is **the presence of a terminal `stop_trigger_raised` for the current dispatch that has not been superseded by a later success/resolution on the same target.** Because the dispatcher writes `stop_trigger_raised` *only* on terminal failure (`retry.py:55` raises ‚Üí `core.py:231` catches ‚Üí `_emit_stop_trigger`), a fail-then-succeed run never produces it (C6) ‚Äî so the negative cell is clean by construction. The subtle case is **AC-3 resume**: a *stale* `stop_trigger_raised` from a prior, since-resolved dispatch must not re-fire. Scope the read to the current dispatch ‚Äî e.g. only consider `stop_trigger_raised` entries that are **more recent than the last `dispatch_attempt(success)` for the same `target_id`**, or that share the iteration's `correlation_id` / are the **most-recent** terminal-vs-success for that target. Prove both directions with the negative + resume cells. The exact rule is **D2**.

## Decisions Needed

- **D1 ‚Äî `reason`-string encoding for `agent` / `attempts` / `last_error` + the debug-output path (C5).** `StopDecision` has only `{fired, trigger, target, reason}`; AC-1 wants `agent=<name>, attempts=3, last_error=<excerpt>` surfaced, and the user "shown the path to the agent's last debug output."
  - **(a) Pack a single human-readable one-liner into `reason`: `agent=<specialist> attempts=<N> last_error=<truncated excerpt> debug=03-Implementation/agent_runs.jsonl`.** `trigger="agent_failed"`, `target=<step/specialist>`. Truncate `last_error` to a fixed budget (e.g. 200 chars) to keep `reason` bounded. The **full** history stays in the journal (`dispatch_attempt` rows + `stop_trigger_raised.last_error`, C1); `reason` is a surfaced summary, not the system of record. The dashboard (5.19) and CLI can re-derive richer detail from the journal. Smallest surface, no new fields, satisfies every AC clause. **(Recommended.)**
  - **(b) Put only `agent=<name> attempts=<N>` in `reason`; rely on the CLI/5.19 banner to render `last_error` + debug path from the journal.** Cleaner `reason`, but AC-1's "last_error=<excerpt>" then has no representation in the StopDecision itself ‚Äî pushes the obligation onto a not-yet-built consumer.
  - **(c) Encode a structured (e.g. `k=v;`-delimited or JSON) blob in `reason`.** Machine-parseable, but `reason` is a free human string elsewhere (4.2 used a bare trigger word), and a JSON-in-string invites brittle parsing; reject in favour of the human one-liner.

- **D2 ‚Äî Supersession / scoping rule for the journal read (C6/C7 ‚Äî makes the negative + resume cells pass).** `check()` must fire on a *current* terminal failure but NOT on a *stale, since-resolved* one.
  - **(a) Most-recent-wins per target: fire iff the latest terminal-relevant entry for the failing `target_id` is `stop_trigger_raised` (i.e. no later `dispatch_attempt(success)` for that target).** Scan `iter_entries` once, track per-`target_id` the last `dispatch_attempt(success)` seq vs the last `stop_trigger_raised` seq; fire on targets whose `stop_trigger_raised` is newer. Pure-fn-of-disk, deterministic (NFR-REL-5), handles resume (a real fix writes a newer success), and is robust to multiple targets. **(Recommended.)**
  - **(b) Correlation-id scoping: only consider entries sharing the current iteration's `correlation_id`.** Tightest scoping to the just-dispatched task, but the loop passes `check_stop` the pre-dispatch `state` (not the `correlation_id`), and `check()`'s Protocol doesn't receive it (frozen) ‚Äî would require re-deriving the latest iteration's correlation_id from the journal anyway, so it collapses into (a) with extra coupling.
  - **(c) "Any `stop_trigger_raised` ever written fires."** Simplest, but **breaks AC-3 resume** (a resolved failure re-halts forever) and **breaks the multi-target case**. Rejected.

- **D3 ‚Äî Priority placement of `AgentFailedTrigger` in `_ORDERED_TRIGGERS` (C2/C7 first-fired).** `check_stop` returns the first-fired decision; ordering only matters if two triggers fire in the same iteration.
  - **(a) Append after the clarification trigger (current tuple order: clarification, then agent_failed).** Clarification is the foundational STOP (4.2); agent-failure is a runtime/execution STOP. Co-firing is unlikely (a clarification halts before dispatch reaches retry-exhaustion), so order is largely cosmetic ‚Äî but a stable, documented order is what the siblings inherit. Append-at-end matches the tuple-growth convention (`stop_registry.py:12`). **(Recommended.)**
  - **(b) Insert at a specific priority index** (e.g. before clarification). No evidence any STOP must out-prioritise another in v1; reordering the frozen tuple invites merge churn with 4.3‚Äì4.8. Defer any real priority contract to a dedicated story if a co-fire case ever surfaces. Reject pre-emptive reordering.
