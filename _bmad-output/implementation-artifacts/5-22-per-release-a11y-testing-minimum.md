## Story

As Murat enforcing per-release a11y discipline (DD-19),
I want the per-release a11y testing minimum codified: axe-core scan as CI gate on every dashboard PR, screen-reader smoke test (NVDA on Windows / VoiceOver on macOS), and keyboard-only smoke test, with results published to release notes,
So that no release ships without explicit a11y verification (UX-DR35, UX-DR36, DD-19, NFR-A11Y-1).

## Acceptance Criteria

> Copied verbatim from `_bmad-output/planning-artifacts/epics.md` (Epic 5, Story 5.22, lines 2889–2907).

**Given** the per-release a11y test harness
**When** a release PR is opened
**Then** axe-core scans the rendered dashboard with full real-data fixtures (post-Story 5.18) and reports zero WCAG 2.2 Level A violations
**And** the keyboard-only test (Story 5.12) passes
**And** the a11y test results are committed to the release notes draft

**Given** a screen-reader smoke test
**When** the release process runs
**Then** a documented manual smoke test checklist is executed (NVDA + VoiceOver covering masthead, KPIs, resume card, phase tracker, backlog tree, STOP banner, activity feed)
**And** the checklist is signed off by a designated reviewer
**And** the signoff is tracked in `docs/a11y/release-<version>.md`

**Given** any a11y regression
**When** the release process runs
**Then** the release is blocked
**And** the regression is filed as a bug ticket
**And** the per-component a11y checklist (UX §8.4) is consulted to identify the failing component

### Decisions (resolve per CONTRIBUTING §5 — record the pick in the PR Change Log)

**D1 — full-surface release fixture vs aggregate existing per-component scans (MED / correctness + scope).** The composite scan omits the STOP banner + disconnection + viewport banner (AC2 names the STOP banner; AC1 wants the full surface). *Recommendation (a):* a committed **full-surface fixture** (`release-a11y-surface.html`) composing existing components incl. a STOP banner + disconnected state + viewport banner, with a terminal axe scan + known-violation witness — catches cross-component landmark/heading-order issues a per-component scan can't, and gives one authoritative release gate. *(b)* aggregate the existing per-component axe results into the report without a new composite — lighter but misses whole-page interactions. *(c)* boot a live real-data `sdlc dashboard` and scan it — **REJECTED** (flaky; index.html isn't assembled; determinism lost). **D2 — release-gate mechanism: report-script + committable artifact vs docs-process-only (MED / process).** AC1 requires a11y results "committed to the release notes draft." *Recommendation (a):* a thin `scripts/generate_a11y_report.py` (reuses `run_axe_scan`) that emits a deterministic report block + exits non-zero on any Level-A violation (the AC3 blocking hook), pasted into `docs/a11y/release-<version>.md`. *(b)* docs-only: rely on the existing pytest axe gate as the block and copy pytest output by hand — "committed to the release notes draft" becomes hand-wavy and unverifiable. Prefer (a), thin. If (a) adds a distinct CI step, wire it into `ci-gate` (§4:Task 4) — a job outside the aggregator is a false green.

**D3 — release-notes/signoff doc location + template-now vs concrete instance (LOW / docs).** No CHANGELOG/releases dir; version `0.0.0` unshipped. *Recommendation (a):* one `docs/a11y/release-<version>.md` per release holds BOTH the axe report and the manual-smoke signoff (matches AC2's named path; avoids inventing a second `docs/releases/`); ship a **`release-template.md`** now — the first concrete `release-<version>.md` is produced at first tag. *(b)* a separate `docs/releases/<version>.md` for notes + `docs/a11y/release-<version>.md` for signoff — two docs, more surface, more drift. **D4 — regression→bug-ticket mechanism + release-blocking enforcement (LOW / process).** *Recommendation (a):* "filed as a bug ticket" = CONTRIBUTING §8 `kind:bug` (cite failing axe rule + reproducer + violated NFR-A11Y); "release is blocked" = the Level-A axe gate already reds `quality-gates` (every PR) + release `qa` (ADR-008) — codify (don't re-invent) this in `docs/a11y/README.md`, cross-linked from CONTRIBUTING §1/§8; §8.4 checklist names the failing component. *(b)* stand up a new external issue-tracker integration — **REJECTED** (YAGNI; §8 already defines the process). Prefer (a).
