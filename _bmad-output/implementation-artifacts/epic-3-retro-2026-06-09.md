## 3. What Went Well (all four endorsed by Project Lead)

1. **The `cli`-injects / `adopt`-core-pure DI pattern — one decision reused five times.** The boundary
   forbids `adopt/` from importing `cli/`, `git`, or `engine/replan`, so nearly every story used the
   same shape: `cli/` reads-and-injects, the `adopt/` core stays pure. 3.8 established it
   (`legacy_code_globs`); 3.2 (git-signal), 3.3 (confirm-callback), 3.6 (conflict-callback) reused it
   verbatim. The single most reused pattern of the epic; it kept the core testable without breaking the
   boundary.
2. **"CODE is ground-truth" — Epic 2B action item A1 (Ground-Truth Recon) internalized.** Grep-verify
   before implementing caught a wrong assumption in every story: confidence is int `[0,100]` not float
   `0.92` (3.2/3.3); the source-untouched standard is `git status --porcelain` + tree-hash not
   `git diff` (3.1); **`sdlc unsign` does not exist** → use the `replan` seam (3.5); **`tdd-strategist`
   already exists** from 2B.10 → author a new specialist instead of overloading (3.8).
3. **Mutation testing surfaced real test rot invisible to all static review.** When 3.7's harness
   finally ran on POSIX it exposed **≥7 mutation-kill tests asserting the *opposite* of production
   behavior** (e.g. `is_target_under_root("")` asserted `True` while production returns `False`). No
   adversarial review layer (Blind Hunter / Edge Case Hunter / Acceptance Auditor) caught these.
4. **Adversarial review caught a CRITICAL data-loss path.** In 3.6 a destructive conflict op (deleting
   the user's real file) was committed *before* the symlink was created, with no compensating journal —
   if create failed, the file was gone, unrecoverable by 3.5 rollback. Review caught it → reordered to
   backup → create → compensate/restore.

## 4. Challenges & Growth Areas (Project Lead: both root causes weighted equally)

**Root cause (one, two altitudes): "commit/test discipline still relied on willpower — no guard."**
A1 (a pre-code practice) internalized beautifully; A2/A3/A4 (which needed an *enforcement mechanism*)
slipped because the guard was never built.

1. **A2 ("done" can't flip while unmerged) was never coded into a guard.** The 2B retro commissioned
   `scripts/check_story_merged_before_done.py`; it **does not exist**. The consequence is visible in
   `git log`: `bc07c23 feat(epic-3): … Stories 3.3–3.4` — **3.3 and 3.4 squashed into one feat commit
   with no separate `test(…) RED`**; `aceabb6 feat(3.5)` single feat commit, no RED; 3.3 and 3.6
   reviewed as uncommitted working-tree blobs.
2. **3.7 — "claim outran evidence" recurred exactly as the 2B retro warned.** Commit
   `5859285 docs(3.7): … status review->done` declared the story DONE while the real kill-rate was
   **70.19%** and on-disk status still read `review` — a **false done**. It took 4 POSIX Docker runs,
   an 8-commit mutation-kill campaign, and a status-integrity correction to actually hit AC2 (96.74%).
3. **The Windows-vs-POSIX blind spot was the epic's defining systemic issue.** The dev host is Windows;
   adopt is POSIX-only (ADR-034), so every adopt test carries a module-level win32 skip. All of 3.1–3.6
   wrote tests that skip on the dev host — real verification was deferred to CI/Docker — and the whole
   module's verification debt concentrated into 3.7, where the mutation suite **had never run on any
   platform** (it errored on collection, then even reported a *false* ~100% from an inherited
   `--cov-fail-under=87`, defect #7) until forced into Docker. A story authored "green on Windows" was
   ~25pp short of its gate.

## 5. Key Insights

- **Hard gates hold; soft gates slip.** A1 internalized because it is natural pre-code work. A2/A3/A4
  slipped for want of a guard. Coverage-90 slipped because it is an "aspiration," not a gate.
- **One process fix attacks both pains:** turn A2/A3 and the POSIX gap into *mechanisms* (a merged-and-RED
  guard + a POSIX pre-merge CI gate) rather than willpower. This is the same conclusion as the 2B retro
  — but now with hard evidence it *must* be coded, not noted.
- **Mutation testing earns its keep** precisely where static review is blind (asserting-the-opposite test
  rot). Worth keeping as a standing gate for security-sensitive modules.
- **Forward-propagated corrections work** (the `**_kw`-absorbing driver-stub warning, CR3.3-P4, carried
  into 3.4/3.6/3.7) — the same "Previous-Story Intelligence" mechanism the 2B retro praised.

## 7. Next Epic Preview — Epic 4: Auto-Mode & Autonomous Execution

`/sdlc-auto` iterates `scan → dispatch → STOP-check` until one of **7 STOP triggers** fires or a
watchdog timeout (default 30 min) expires; auto-brainstorm panel surfaces options-with-tradeoffs
(framework never picks); mad-mode `/sdlc-auto-mad` auto-resolves some STOPs, reversible via
`sdlc unsign --mad-only`. Per Murat's risk lens, **1 STOP trigger = 1 story** with a 4-cell matrix
(positive / negative / termination / resume) = ≥28 test cells. New modules: `engine/auto_loop.py`,
`engine/stop_triggers.py`, `engine/auto_brainstorm.py`, `engine/auto_mad.py`.

**Dependencies:** Epic 4 builds chiefly on the Epic 1/2A substrate (dispatcher, journal, engine), **not
on adopt** — so Epic 3 barely blocks it. A pre-wired seam already exists:
`dispatcher.core._emit_stop_trigger()` (2A.3) writes `kind=stop_trigger_raised` with
`epic_4_placeholder=True`; Story 4.6 consumes it (`EPIC-4-STOP-TRIGGER-WIRE`).

**Two Epic-3 carry-ins to assess before Story 4.1:**
1. **`sdlc unsign --mad-only` (FR23, story 4.12) is a net-new command.** Story 3.5 confirmed
   `sdlc unsign` does not exist — the realised seam is the `replan` pattern (`invalidate_record` +
   `signoff_invalidated`). 3.5 already mapped it, so Epic 4 need not re-discover it.
2. **`EPIC-3-DEBT-CHARACTERIZATION-REAL-DISPATCH` (3.8) + `EPIC-2A-DEBT-TASK-REAL-TEST-EXECUTION`.** Real
   dispatch is currently *nominal-only* (always dispatches `test-author` → RED). Epic 4's auto-loop **is**
   real dispatch, so this debt may bite — and it is **not yet in `debt-budget.yaml`** (a tracking gap).
